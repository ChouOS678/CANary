# 项目简介

本项目是一个 **CAN 总线异常检测系统**，用于识别汽车 CAN（Controller Area Network）总线通信中的多种攻击行为。系统在稳态巡航场景中，通过机器学习方法自动检测夹杂在正常通信流量中的异常插针攻击。

### 检测的攻击类型

| 类别 | 说明 |
|------|------|
| 安全 | 正常 CAN 总线通信流量 |
| DoS | 拒绝服务攻击，表现为高流量、高突发性和高错误率 |
| 重放 | 攻击者捕获并重放合法报文 |
| 模糊 | 向总线注入随机/模糊数据帧 |
| 欺骗 | 伪造 ECU 身份发送欺骗性报文 |
| UDS非法会话 | 利用 UDS 诊断协议建立非法会话 |

### 技术架构

- **模型**: 随机森林分类器（RandomForestClassifier），基于 scikit-learn，含 5 折分层交叉验证
- **风险评估**: 基于领域知识的 5 组加权指标线性组合（流量速率/错误率/突发性/时间间隔/攻击类型），参考 NIST SP 800-61 分级思想
- **特征工程**: 提取 11 个 CAN 流量特征，包括流量速率、帧率、ID 熵、攻击比率、突发分数等
- **数据生成**: 合成数据生成器，支持边界样本、漂移样本和标签噪声，模拟真实攻击场景
- **可视化**: Matplotlib 静态图表 + Plotly 交互式图表双引擎
- **Web 界面**: 基于 Streamlit 的可视化系统，支持参数调节、模型缓存、实时预测和性能对比
- **性能分析**: 集成 Intel VTune Profiler 硬件 PMC 事件采集，可获取真实 CPU 缓存命中率数据
- **单元测试**: pytest 框架，覆盖配置校验、数据生成、模型训练/预测全链路（59 个测试用例）

### 多变量维度对比分析

`perf_compare.py` 从多个独立维度对两种算法进行对比，确保分析的全面性与客观性：

**维度一：端到端训练/预测耗时**
- 统一计时口径：包含数据加载 + 训练 + 预测全流程，直方图算法额外包含 float64→uint8 编码耗时
- 在相同硬件、相同数据下同时运行两算法，确保公平性

**维度二：CPU 硬件利用率**
- 通过 `psutil` 实时采样 CPU 总利用率（均值/峰值）和逐核利用率
- 生成逐核热力图，可视化各核心负载分布

**维度三：数据内存占用与缓存驻留**
- 确定性计算：`N × 11 × sizeof(dtype)`
- 对比 float64（8 字节/特征）vs uint8（1 字节/特征）的数据集内存占用
- 标注 L1D Cache（32KB）阈值线，判断数据是否可驻留在最高速缓存

**维度四：随机行访问基准测试（timeit）**
- 模拟决策树训练的核心访问模式 `mat[random_indices]`
- 多规模测试（1K / 10K / 50K / 100K 样本）
- 避免使用 sum/sort 等 SIMD 优化操作（不反映真实训练访问模式）

**维度五：控制变量隔离分析**
- 同时改变 dtype 和内存布局会混淆因果，故采用"三组实验、每组只改一个变量"
- 固定 float64 → 改 order C vs F：隔离内存布局的独立影响
- 固定 C-order → 改 dtype float64 vs uint8：隔离数据类型的独立影响
- 固定 uint8 → 改 order C vs F：验证内存布局影响的一致性

**维度六：数据规模扩展性**
- 从 1K 到 500K 样本量，展示两种算法数据量的对数增长趋势
- 标注 L1D(32KB) / L2(256KB) 缓存阈值线
- 清晰展示 scikit-learn 何时超出各缓存层级

## 算法分析与优化及验证

### 传统 scikit-learn 算法的局限性

本项目初始版本采用 scikit-learn 的 `RandomForestClassifier` 直接对原始 float64 特征值进行训练。在功能正确的前提下，该方案存在以下性能瓶颈：

| 问题维度 | 具体表现 |
|----------|----------|
| **内存占用大** | 每个特征值使用 float64（8 字节），11 个特征每样本占 88 字节；1920 样本即达 165 KB，已超出 L1D 缓存（32 KB） |
| **缓存不友好** | sklearn 内部决策树构建时按随机索引访问样本，每次访问均需从 L2/L3 甚至主存搬运 88 字节/行 |
| **Memory Wall 效应** | 数据量增大后（>10K 样本），float64 矩阵远超 L1/L2 缓存容量，CPU 大量时钟周期浪费在等待数据搬运而非实际计算 |
| **扩展性差** | 当样本量达到 100K 时，数据量达 8.4 MB，只能驻留主存；缓存未命中率急剧上升，性能严重退化 |

### 直方图离散化算法的设计动机

针对上述问题，本项目引入**直方图离散化算法**（`histogram_model.py`），核心思路是：

1. **特征离散化**：将连续 float64 特征值映射到 256 个等宽桶中，用 uint8（1 字节）表示
2. **内存缩减 87.5%**：每样本从 88 字节降至 11 字节，同等规模数据量仅为原来的 1/8
3. **缓存驻留优势**：uint8 矩阵在更大数据规模下仍可驻留 L1D 缓存，减少 CPU 等待时钟
4. **带宽优势**：决策树随机查找同等数量的样本时，uint8 仅需搬运 1/8 的数据量，有效缓解 Memory Wall

### 实验验证

通过 `perf_compare.py` 模块进行多规模随机行访问基准测试（模拟决策树训练中的样本查找模式），实验结果如下：

**数据随机访问耗时对比（timeit，30 次迭代）**

| 样本量 | scikit-learn (float64) | 直方图 (uint8) | 加速比 |
|-------:|----------------------:|----------------:|-------:|
| 1,000 | 0.0141 ms (86 KB) | 0.0082 ms (11 KB) | **1.71×** |
| 10,000 | 0.0216 ms (859 KB) | 0.0172 ms (107 KB) | **1.25×** |
| 50,000 | 0.0234 ms (4.2 MB) | 0.0167 ms (537 KB) | **1.40×** |
| 100,000 | 0.0251 ms (8.4 MB) | 0.0169 ms (1.0 MB) | **1.49×** |

**缓存层级对比（随数据规模变化）**

| 样本量 | scikit-learn 缓存层级 | 直方图算法缓存层级 |
|-------:|:-------------------:|:----------------:|
| 1,000 | L2 (86 KB) | **L1D** (11 KB) |
| 5,000 | L3 (430 KB) | L2 (54 KB) |
| 10,000 | L3 (859 KB) | L2 (107 KB) |
| 50,000 | L3 (4.2 MB) | L3 (537 KB) |
| 100,000 | **主存** (8.4 MB) | L3 (1.0 MB) |
| 500,000 | **主存** (42 MB) | L3 (5.2 MB) |

**控制变量隔离分析（1920 样本，随机行访问）**

| 对比维度 | 结果 |
|----------|------|
| 固定 float64，改内存布局 C→F | F-order 慢 1.24×（非连续内存增加缓存未命中） |
| 固定 C-order，改数据类型 float64→uint8 | uint8 快 1.41×（1/8 带宽的直接优势） |
| 固定 uint8，改内存布局 C→F | F-order 慢 1.29×（同样的缓存局部性影响） |

### Intel VTune 硬件缓存实测

缓存命中率对比是本项目性能分析的核心。为获取模型在真实硬件上的缓存行为数据，项目集成了 **Intel VTune Profiler** 进行硬件 PMC（Performance Monitoring Counter）事件采集，从根本上替换了之前的理论模型 `min(1.0, C/D)`。

**为什么要用 VTune？**
- 理论模型无法反映乱序执行、硬件预取、缓存替换策略等微架构细节
- VTune 直接读取 CPU 硬件 PMC 寄存器，获取真实的 MEM_LOAD_RETIRED 事件计数
- 只有硬件实测数据才能准确反映两种算法在 i5-13500H 上的真实缓存表现

**采集方案**

| 项目 | 说明 |
|------|------|
| **工具** | Intel VTune Profiler 2026.2.0 (build 632324) |
| **分析类型** | uarch-exploration（微架构探索分析） |
| **硬件事件** | MEM_LOAD_RETIRED.{L1_HIT, L2_HIT, L3_HIT, L3_MISS} 等 150+ 种 PMC 事件 |
| **采集组合** | sklearn/histogram × 5K/20K 样本（4 组） |
| **数据存储** | `output/vtune_results/cache_metrics.json` |
| **采集脚本** | `_vtune_collect.bat`（需管理员权限运行） |

**实测结果（20K 样本，i5-13500H Raptor Lake-P）**

| 指标 | scikit-learn (float64) | 直方图 (uint8) |
|------|----------------------:|----------------:|
| L1D 命中率 | 97.18% | 95.24% |
| L2 命中率 | 77.78% | 82.35% |
| L3 命中率 | 37.51% | **66.68%** |
| L3 Miss 计数 | **16,500,000** | 6,600,000 |

> 🔑 关键发现：sklearn 的 L3 Miss 是直方图的 **2.5 倍**，意味着 sklearn 的 CPU 需要更多次访问 DRAM（延迟 ~100ns vs L3 ~15ns），这是直方图算法性能优势的**硬件级根因**。

**数据集成逻辑**（`perf_compare.py`）
- `load_vtune_cache_metrics()`：从 JSON 加载实测事件计数
- `cache_analysis()`：从 PMC 计数计算 L1/L2/L3 命中率
- 精确匹配 → 比例推算 → 理论模型回退（三级优先策略）
- 所有图表和报告自动标注数据来源："Intel VTune PMC 实测" / "理论模型"

### 结论

- **小规模数据**（<2K 样本）：两种算法端到端耗时差异有限，数据仍可被 L2/L3 缓存覆盖，Memory Wall 效应尚不显著
- **中大规模数据**（>10K 样本）：scikit-learn 的 float64 数据迅速超出高速缓存容量，触发严重的计算与数据供给速度冲突；直方图算法凭借 1/8 的内存占用，在同等规模下仍可驻留更高层级缓存，随机访问速度快 **1.25~1.71×**
- **实际意义**：对于 CAN 总线等高吞吐场景，直方图离散化算法在保持 256 桶精度（对决策树分裂点选择影响极小）的同时，在数据访问效率上具有显著且可扩展的优势

> 注：numpy 的 SIMD 内核对 float64 向量化运算（sum/sort）高度优化，在纯计算操作上 float64 可能更快。但决策树训练的核心瓶颈是随机样本查找（非向量化操作），此处 uint8 的内存带宽优势起决定性作用。

## 模型验证与质量保障

### 风险评分权重系数的理论依据

`model.py` 中的 `compute_case_metrics` 函数使用 5 组加权指标的线性组合计算风险分数，所有权重和为 1.0，每项归一化至 [0, 1]：

| 指标组 | 权重 | 归一化方式 | 设计依据 |
|--------|:----:|-----------|----------|
| **flow_kbps_mean** | 0.28 | ÷ 560 kbps（上界） | DoS 攻击首要特征，流量激增是 CAN 总线最直接的物理层异常信号 |
| **error_ratio** | 0.16 | ÷ 0.35（上界） | 模糊攻击和 DoS 均导致总线错误率上升；权重适中（正常竞争也会产生错误） |
| **burst_score** | 0.18 | 直接使用（已在 [0.05, 0.98]） | 攻击报文以突发方式注入，与正常周期性 CAN 报文形成对比 |
| **inter_arrival_cv** | 0.14 | 直接使用（已在 [0.05, 0.95]） | 攻击注入破坏周期性规律；权重较低（网络抖动也可引起类似变化） |
| **max(四种攻击比率)** | 0.24 | 直接使用（各比率在 [0, 0.95]） | 四种攻击类型（重放/模糊/欺骗/UDS）共享此权重；使用 max() 而非 sum() 避免多类型叠加场景的过度敏感 |

**风险分数合成逻辑**：
- 预测为「安全」时：`(1-safe_probability) × 0.32 + feature_attack_score × 0.28`（模型置信度权重更高）
- 预测为攻击时：`confidence × 0.56 + feature_attack_score × 0.44`（≈1.27:1 反映模型判断主导性）

**严重度分级**（仅攻击预测生效，参考 NIST SP 800-61 事件分级思想）：

| risk_score | 高:中:低 分流比 | 含义 |
|:----------:|:--------------:|------|
| ≥ 0.8 | 49:33:18 | 高风险场景，大部分异常包被标记为高危 |
| ≥ 0.58 | 28:44:28 | 中等风险，呈正态分布 |
| < 0.58 | 48:34:18 | 低风险，多数为低等级 |

### 交叉验证

`train_model` 在训练最终模型前，先对训练集执行 **5 折分层交叉验证**（`StratifiedKFold`），提供更稳健的模型泛化能力评估：

- **目的**: 单次 hold-out 划分可能因数据分割的偶然性导致评估偏差，k-fold CV 通过多次划分取均值获得更稳定的估计
- **分层策略**: 保持每折中各类别标签比例与原始数据一致，避免小类被遗漏
- **输出指标**: `cv_mean_accuracy`（均值）`cv_std_accuracy`（标准差）`cv_fold_accuracies`（各折准确率列表）
- **呈现位置**: 终端输出、`run_summary.json`、`run_summary.txt` 均包含 CV 结果

运行示例输出：
```
交叉验证准确率 (5-fold)：0.9702 ± 0.0018
```

### 报告内容

| 章节 | 内容 |
|------|------|
| **封面** | 标题、副标题（scikit-learn float64 vs 直方图 uint8） |
| **测试配置** | 样本总量、特征维度、每标签样本数 |
| **核心性能指标** | 训练/预测耗时、加速比、准确率对比 |
| **内存与缓存** | 数据量对比、VTune PMC 实测 L1D/L3 命中率与 Miss 计数 |
| **可视化图表** | 耗时对比、缓存命中率、CPU 利用率、综合效能雷达图（4 张） |
| **分析结论** | 完整的算法效能对比分析文本 |

### 技术实现

- **报告引擎**: [fpdf2](https://github.com/py-pdf/fpdf2)（MIT 协议，支持 Unicode 中文渲染）
- **中文字体**: 自动检测系统 SimHei / Microsoft YaHei / Noto Sans CJK
- **图表嵌入**: 将 Matplotlib 生成的 PNG 图表直接嵌入 PDF 页面
- **缓存指标**: 优先使用 Intel VTune PMC 硬件实测数据；无实测数据时回退理论模型
